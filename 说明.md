## 一、Skills 工具的安装流程全景

整体流程可以概括为 **4 个阶段**：

```
用户输入源 → 解析来源 → 克隆/读取仓库 → 发现 SKILL.md → 安装到 Agent 目录
```

具体地说：

1. `source-parser.ts` 解析用户输入，判断来源类型（GitHub / GitLab / Git SSH / 本地路径 / Well-Known URL）
2. `git.ts` 通过 `git clone --depth 1` 克隆仓库到临时目录（本地路径则直接使用）
3. `skills.ts` 的 `discoverSkills()` 在克隆下来的目录中**按优先级搜索** `SKILL.md` 文件
4. `installer.ts` 将发现到的 skill 目录复制到 canonical 位置（`.agents/skills/<skill-name>/`），然后为各 Agent 创建 symlink

## 二、核心协议：SKILL.md 规范

这是整个系统的**唯一核心约定**——一个 skill 就是一个**包含 `SKILL.md` 文件的目录**。

### 必须遵守的规则

`SKILL.md` 文件必须包含 YAML frontmatter，其中有两个**必填字段**：

```markdown
---
name: my-skill-name
description: 这个 skill 做什么、什么时候使用它
---

# 正文内容（给 Agent 看的指令）

具体的指令、步骤、规则等...
```

关键约束来自 `skills.ts` 的 `parseSkillMd()`：

```28:43:src/skills.ts
export async function parseSkillMd(
  skillMdPath: string,
  options?: { includeInternal?: boolean }
): Promise<Skill | null> {
  try {
    const content = await readFile(skillMdPath, 'utf-8');
    const { data } = matter(content);

    if (!data.name || !data.description) {
      return null;
    }

    // Ensure name and description are strings (YAML can parse numbers, booleans, etc.)
    if (typeof data.name !== 'string' || typeof data.description !== 'string') {
      return null;
    }
```

规则总结：

| 要求 | 说明 |
|------|------|
| 文件名 | 必须叫 `SKILL.md`（大写） |
| `name` | 必填，必须是字符串，作为 skill 的唯一标识符 |
| `description` | 必填，必须是字符串，描述 skill 用途 |
| `metadata.internal` | 可选，设为 `true` 时隐藏此 skill（除非设 `INSTALL_INTERNAL_SKILLS=1`） |

安装时 `name` 会被 `sanitizeName()` 处理成 kebab-case 作为目录名：

```39:54:src/installer.ts
export function sanitizeName(name: string): string {
  const sanitized = name
    .toLowerCase()
    .replace(/[^a-z0-9._]+/g, '-')
    .replace(/^[.\-]+|[.\-]+$/g, '');

  return sanitized.substring(0, 255) || 'unnamed-skill';
}
```

## 三、Skills 存放规则——工具如何发现它们

`discoverSkills()` 有一个**分层搜索策略**，按以下优先级顺序扫描：

### 第一优先级：直接指向

如果用户通过 URL 子路径（如 `/tree/main/skills/my-skill`）直接指向了一个包含 `SKILL.md` 的目录，就直接使用它，不再继续搜索。

### 第二优先级：约定目录（推荐使用）

工具会搜索以下**约定目录的直接子目录**中的 `SKILL.md`：

```153:184:src/skills.ts
  const prioritySearchDirs = [
    searchPath,
    join(searchPath, 'skills'),
    join(searchPath, 'skills/.curated'),
    join(searchPath, 'skills/.experimental'),
    join(searchPath, 'skills/.system'),
    join(searchPath, '.agent/skills'),
    join(searchPath, '.agents/skills'),
    join(searchPath, '.claude/skills'),
    // ... 更多 agent 特定目录
  ];
```

**重点**：工具搜索的是这些目录的**一级子目录**。也就是说，skill 必须是这些路径下的**直接子文件夹**，而不能嵌套更深。

例如 `skills/my-skill/SKILL.md` 会被发现，但 `skills/category/my-skill/SKILL.md` 在优先级搜索中**不会**被发现。

### 第三优先级：递归搜索（兜底）

如果上述约定目录都没找到任何 skill，工具会进行最大深度为 5 的递归搜索：

```211:223:src/skills.ts
  if (skills.length === 0 || options?.fullDepth) {
    const allSkillDirs = await findSkillDirs(searchPath);

    for (const skillDir of allSkillDirs) {
      let skill = await parseSkillMd(join(skillDir, 'SKILL.md'), options);
      if (skill && !seenNames.has(skill.name)) {
        skill = enhanceSkill(skill);
        skills.push(skill);
        seenNames.add(skill.name);
      }
    }
  }
```

递归搜索会跳过 `node_modules`、`.git`、`dist`、`build`、`__pycache__` 等目录。

## 四、支持的源格式（source-parser.ts）

工具支持非常灵活的来源指定方式：

| 格式 | 示例 | 类型 |
|------|------|------|
| GitHub 简写 | `owner/repo` | `github` |
| GitHub URL | `https://github.com/owner/repo` | `github` |
| GitHub 子路径 | `https://github.com/owner/repo/tree/main/path` | `github` |
| GitLab URL | `https://gitlab.com/group/repo` | `gitlab` |
| GitLab 子组 | `https://gitlab.com/group/subgroup/repo` | `gitlab` |
| GitLab 子路径 | `https://gitlab.example.com/group/repo/-/tree/main/path` | `gitlab` |
| 前缀简写 | `gitlab:group/repo` | `gitlab` |
| Git SSH | `git@gitlab.example.com:group/repo.git` | `git` |
| 本地路径 | `./my-skills` | `local` |
| 任意 HTTP(S) URL | `https://custom-server.com` | `well-known` |

对于**私有 GitLab**，关键代码在：

```196:240:src/source-parser.ts
  // GitLab URL with path (any GitLab instance): https://gitlab.com/owner/repo/-/tree/branch/path
  const gitlabTreeWithPathMatch = input.match(
    /^(https?):\/\/([^/]+)\/(.+?)\/-\/tree\/([^/]+)\/(.+)/
  );
  // ...
  // GitLab.com URL: https://gitlab.com/owner/repo or https://gitlab.com/group/subgroup/repo
  const gitlabRepoMatch = input.match(/gitlab\.com\/(.+?)(?:\.git)?\/?$/);
```

注意：对于**非 gitlab.com 的自建 GitLab 实例**，需要使用 `/-/tree/` 路径格式或 Git SSH URL 才能被正确识别。如果只提供 HTTPS URL 且不包含 `gitlab.com` 域名，则会被当作 `well-known` 类型或 `git` 类型处理——但 `git clone` 依然可以正常工作。

## 五、安装机制详解

### 安装位置

| 作用域 | 位置 | 说明 |
|--------|------|------|
| 项目级（默认） | `./<agent>/skills/<skill-name>/` | 随项目提交 |
| 全局级（`-g`） | `~/<agent>/skills/<skill-name>/` | 跨项目可用 |

### Symlink 模式（默认推荐）

```
.agents/skills/<skill-name>/     ← 规范位置（canonical），实际文件在这里
.claude/skills/<skill-name>/     ← symlink → .agents/skills/<skill-name>/
.cursor/skills/<skill-name>/     ← symlink → .agents/skills/<skill-name>/（如果不是 universal agent 的话）
```

Cursor 等 "universal agent" 本身就使用 `.agents/skills/` 路径，所以不需要额外的 symlink。

## 六、私有 GitLab 实例的实操指南——以 `gubo-common-skills` 仓库为例

### 前提条件

你的机器需要配置好对私有 GitLab 的 Git 访问权限（SSH key 或 HTTPS 凭据），因为工具底层是通过 `git clone --depth 1` 来获取仓库的。

### 推荐的仓库目录结构

假设公司 GitLab 地址是 `https://gitlab.gubo.com`，仓库为 `gubo/gubo-common-skills`。

如果需要**按前端/后端等分类**组织 skills，推荐使用 `skills/` 作为统一入口目录，并利用**插件清单（Plugin Manifest）** 机制实现分组：

```
gubo-common-skills/
├── .claude-plugin/
│   └── marketplace.json          ← 可选：用于 skill 分组展示
├── skills/
│   ├── frontend-design/
│   │   └── SKILL.md
│   ├── frontend-component-guide/
│   │   └── SKILL.md
│   ├── frontend-testing/
│   │   └── SKILL.md
│   ├── backend-api-design/
│   │   └── SKILL.md
│   ├── backend-database/
│   │   └── SKILL.md
│   ├── backend-error-handling/
│   │   └── SKILL.md
│   ├── devops-ci-cd/
│   │   └── SKILL.md
│   └── common-code-review/
│       └── SKILL.md
└── README.md
```

**为什么 skills 都平铺在 `skills/` 一级子目录下？** 因为工具的优先级搜索只扫描 `skills/` 的**直接子目录**。如果你把 skill 放到 `skills/frontend/design/SKILL.md`，优先级搜索阶段找不到它（只有递归搜索兜底才会发现）。

### 通过 Plugin Manifest 实现分组显示

如果想让安装时 skill 按"前端/后端/DevOps"分组显示，可以创建 `.claude-plugin/marketplace.json`：

```json
{
  "metadata": { "pluginRoot": "." },
  "plugins": [
    {
      "name": "frontend",
      "source": ".",
      "skills": [
        "./skills/frontend-design",
        "./skills/frontend-component-guide",
        "./skills/frontend-testing"
      ]
    },
    {
      "name": "backend",
      "source": ".",
      "skills": [
        "./skills/backend-api-design",
        "./skills/backend-database",
        "./skills/backend-error-handling"
      ]
    },
    {
      "name": "devops",
      "source": ".",
      "skills": [
        "./skills/devops-ci-cd"
      ]
    }
  ]
}
```

这样在运行 `skills add` 时，交互界面会按 **Frontend / Backend / DevOps** 分组展示，而非一个扁平列表。

以 `skills/frontend-design/SKILL.md` 为例：


### 安装命令

```bash
# 方式一：Git SSH URL（推荐用于私有仓库）
npx skills add git@gitlab.gubo.com:gubo/gubo-common-skills.git

# 方式二：HTTPS URL
npx skills add https://gitlab.gubo.com/gubo/gubo-common-skills.git

# 列出仓库中所有可用的 skills
npx skills add git@gitlab.gubo.com:gubo/gubo-common-skills.git --list

# 只安装特定 skill
npx skills add git@gitlab.gubo.com:gubo/gubo-common-skills.git --skill frontend-design

# 安装到指定 agent，全局安装，跳过确认
npx skills add git@gitlab.gubo.com:gubo/gubo-common-skills.git -a cursor -g -y

# 安装所有 skills 到所有 agents
npx skills add git@gitlab.gubo.com:gubo/gubo-common-skills.git --all

# 安装特定子路径下的 skill
npx skills add https://gitlab.gubo.com/gubo/gubo-common-skills/-/tree/main/skills/frontend-design
```

### 关键注意事项

1. **Git 认证**：私有 GitLab 需要提前配好 SSH key 或 HTTPS 凭据，否则 `git clone` 会失败（超时60秒后报错）
2. **自建 GitLab 域名**：对于非 `gitlab.com` 的自建实例，使用 Git SSH URL（`git@...`）是最可靠的方式，因为 source-parser 只对 `gitlab.com` 做了域名级别的 GitLab 类型识别；其他域名的 HTTPS URL 会走 `git` 或 `well-known` 类型，但 clone 本身不受影响
3. **扁平结构**：所有 skill 目录必须是 `skills/` 的**直接子目录**，不要再嵌套分类文件夹（分类通过 `marketplace.json` 实现）
4. **name 唯一性**：每个 skill 的 `name` 字段在仓库内必须唯一，工具通过 `seenNames` Set 去重

以上就是从源码角度梳理出的完整机制。核心要点再概括一下：

- **协议极简**：一个 skill = 一个目录 + 一个带 `name`/`description` frontmatter 的 `SKILL.md` 文件
- **发现靠约定**：放在 `skills/` 一级子目录下是最可靠的被发现方式
- **分类靠清单**：不要用嵌套文件夹分类，而是用 `.claude-plugin/marketplace.json` 声明分组
- **私有仓库靠 SSH**：对自建 GitLab，`git@` 开头的 SSH URL 是最稳妥的安装方式
